SiK-LSS
Speed is King —
Legend-Signal-System

What I found, what I tested, and what I think it means for agent pipelines. Make of it what you will.

⚡ USPTO Provisional #64,014,841  ·  Filed March 23, 2026  ·  EtherHall LLC
Everything here is my understanding — Jacob Gibson — translated using my own AI workflow. My words. My discovery. AI helped me render it legible. I tried hard not to bias the question when I was verifying this. Hope but verify. That's the whole doctrine.

What I observed.

Every agentic AI pipeline I looked at asks the model to produce structured data — usually JSON — so the system knows what action to take. The model wraps it in code fences. The system strips the fences. Validates the schema. Retries when it fails.

That whole layer exists because the model keeps getting it wrong.

If you already know what the possible actions are — why are you asking the model to type them back out to you?

That question led me somewhere. The model doesn't need to produce JSON at the decision boundary. It needs to signal intent. One character. The system owns all the execution details. The parsing layer has nothing to parse.

What I tested.

I also asked multiple AI systems — with no LSS context and no leading prompts — what eliminating the JSON parsing layer would mean for agentic pipelines. They all independently called it significant. That's not proof. It's the kind of thing that makes you keep going.

What I think it could mean.

If you're building agents and you're maintaining a parsing and validation layer to handle model output — this might be worth looking at. The data suggests the layer isn't necessary if you stop asking the model to produce structured output at the decision boundary in the first place.

I'm not going to overstate this. What I can say is that in my testing it worked cleanly across multiple models. Whether it holds up in your pipeline at your scale — only you can verify that. Hope but verify.

Method · C1–C4

How it actually works.

Four conditions. All four have to be true simultaneously. That's the whole thing.

C1 — Single Token

The model's entire action output at each decision step is exactly one token. Not a word. Not JSON. One character drawn from the legend.

C2 — Legend Once

The symbol table is injected once at session initialization. It is never re-injected on subsequent calls. The model retains it.

C3 — System Owns Execution

The model never outputs tool names, arguments, URLs, file paths, or query strings. The system builds those from its own state machine.

C4 — No Normalization

The token is natively parseable. No fence stripping. No schema validation. No retry logic. The parse layer has nothing to parse.

Here's the minimal proof of concept. Three pieces: the legend in the system prompt, one decision step, one dispatch.

Step 1 — Inject the legend once (system prompt)
# System prompt — sent exactly once at session initialization

You are an autonomous research agent.

LEGEND: S=web_search  F=fetch_page  R=read_memory  W=write_memory  D=done

At each step you will receive current context.
Respond with exactly one character from the legend on the first line.
Brief intent on the second line (for logging — not used for dispatch).

Example:
S
search for recent papers on mixture-of-experts scaling
Step 2 — The decision step (what actually crosses the boundary)
System
Goal: summarize the state of MoE scaling research 2025–2026
Last result: [none]
Next action?
Model
S
search for mixture-of-experts scaling 2025
System
[parses first char — no schema required]
[dispatches web_search — system builds the query string]
[returns results to model context]
Step 3 — Dispatch (Python, ~10 lines)
# The entire parsing and dispatch layer

dispatch = {
    "S": lambda: web_search(build_query(state)),
    "F": lambda: fetch_page(state["last_url"]),
    "R": lambda: read_memory(state["goal"]),
    "W": lambda: write_memory(state["last_result"]),
    "D": lambda: done(state["history"]),
}

# Decision step
response  = call_model(context, max_tokens=1)   # hard cap at 1
symbol    = response.strip()[0]                    # first char — that's it
result    = dispatch[symbol]()                       # system builds every arg

That max_tokens=1 on the API call is doing real work. It's not convention — it's enforcement. The model cannot produce JSON it isn't allowed to finish. The parse layer disappears because there's nothing structurally complex enough to fail.

The key insight is where the query string comes from. In the dispatch above, build_query(state) is a deterministic function in your system — it reads the mission goal and current context board. The model never typed a query string. It never had the chance to hallucinate one, inject into it, or format it wrong. The model said S. Your system did the rest.

That's the architectural shift. Most agent pipelines ask the model to produce structured output and then build infrastructure to clean it up when it doesn't. LSS treats the model as an intent classifier and keeps execution details in deterministic system code. The format drift problem doesn't get solved — it gets removed from the design.

The comparison baseline: a JSON decision step on a 7B model on equivalent hardware produced 0% valid output without defensive infrastructure across 25 trials. The same model with the two-line symbol format: 100% across 25 trials. Zero model changes. Zero hardware changes. The failure was the schema requirement.

If you want to run this against your own pipeline: swap one decision step. Replace your JSON prompt with a legend + single-char constraint. Set max_tokens=1. Move your tool argument construction into a resolver function that reads from your existing state. Count your retries before and after. That's the test.
